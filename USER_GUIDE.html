1. What Is Rosetta Cards MCP?

Rosetta Cards MCP is a knowledge management system built as a Model Context Protocol (MCP) server. It turns text, documents, and operational data into structured, versioned, content-addressed artifacts that can be searched, bundled, and shared.

It works in two ways:

  • As an MCP server — an AI assistant (Claude, or any MCP-compatible client) calls its tools to store, search, and manage knowledge.
  • As a terminal application (TUI) — a human opens the interactive interface to browse cards, import documents, create bundles, and manage behavior packs.

Every artifact is assigned a SHA-256 identity hash computed from its content. If two artifacts have identical content, they get the same hash — there are never duplicates. Timestamps and provenance metadata are excluded from the hash, so the same knowledge always produces the same ID regardless of when or where it was created.

2. Installation and First Run

Requirements

  • Node.js 20 or newer (ESM modules and modern APIs)
  • npm (ships with Node.js)

Install

git clone <repository-url> rosetta-cards-mcp
cd rosetta-cards-mcp
npm ci

Verify everything works

npm test          # run the test suite
npm run build     # compile TypeScript
npm run smoke     # deterministic hash validation

Start the TUI

npm run tui

Start the MCP server

npm run dev       # development mode (ts-node, no build step)
npm run start     # production mode (requires npm run build first)

The MCP server communicates over stdio. Connect your MCP client to it.

3. Core Concepts

3.1 Artifacts

An artifact is any piece of knowledge the system stores. Every artifact has:

FieldDescription
idSHA-256 hash of the artifact's structural content
versionSchema version (currently artifact_v1)
kindOne of: fact, event, decision, skill, profile, tool_obs, summary, project
payloadThe actual content (arbitrary JSON object)
tagsLabels that are part of the artifact's identity
refsReferences to other artifacts ([{kind, id}])
created_atWhen the artifact was first stored (not part of the hash)
sourceProvenance info: agent, tool, repo (not part of the hash)

Artifact kinds explained

  • fact — A factual claim. Example: “The boiling point of water is 100°C.”
  • event — Something that happened at a point in time. Example: “Deployment to production at 3pm.”
  • decision — A choice that was made. Example: “We chose PostgreSQL over MySQL.”
  • skill — A procedure or capability. Example: “How to deploy a Docker container.”
  • profile — A description of an entity. Example: “Alice — backend engineer, Python specialist.”
  • tool_obs — An observation from a tool. Example: “CPU usage hit 95% at 14:32.”
  • summary — A condensed view of other artifacts. Example: weekly summary.
  • project — Project-level metadata. Example: “Project Alpha — Q3 2026 roadmap.”

3.2 Content Addressing

Every artifact gets its identity from a SHA-256 hash of its structural content. The hash is computed from these fields only:

  • version (always “artifact_v1”)
  • kind
  • payload
  • tags (sorted alphabetically)
  • refs

These fields are excluded from the hash:

  • created_at and last_seen_at (timestamps)
  • source (provenance — who/what created it)
Result: Storing the same content twice returns the same ID (deduplication). You can re-store an artifact from a different machine and get the same hash.

Prohibited payload keys: To prevent accidental non-determinism, the system rejects payloads containing keys like created_at, timestamp, hostname, pid, cwd, env, or prototype pollution vectors (__proto__, constructor).

3.3 The Vault

The vault is the storage layer. It stores artifacts in two places simultaneously:

  1. Blob files on disk — each artifact is saved as a JSON file at .vault/blobs/<first-2-chars>/<next-2-chars>/<full-hash>.json. This sharded directory structure keeps any single folder from getting too large.
  2. SQLite index — a database at .vault/index.sqlite provides full-text search (FTS5) over artifact content. WAL mode for concurrent access.
  3. Embeddings database (optional) — at .vault/embeddings.sqlite, stores vector embeddings for semantic search. Requires a running embedding endpoint.

The vault root defaults to .vault/ in the project directory. Override with the ARTIFACT_VAULT_ROOT environment variable.

3.4 Knowledge Cards

A knowledge card is a visual representation of a knowledge artifact:

  • card.json — structured data with title, bullet points, tags, diagram, and sources
  • card.png — a 1200×675 infographic image, optionally with a QR code

Cards are built from documents. When you add a document, it gets chunked into paragraphs. You can then build a card from any chunk.

3.5 NFT Tiers

Three access tiers, designed for ThreadForge’s NFT-based access model. Default is Gold (unrestricted).

TierArtifact CapAllowed KindsTool Access
Bronze 1,000 fact, event, tool_obs Vault, memory, basic KB
Silver 10,000 All kinds Bronze + promotion, blessing, corpus, execution, storage
Gold Unlimited All kinds Everything (no restrictions)

Set via THREADFORGE_TIER environment variable. Case-insensitive. Invalid values fall back to gold.

5. MCP Tools Reference

These tools are available when the system runs as an MCP server. An AI assistant or MCP client calls them by name with JSON arguments.

5.1 vault.* — Artifact Vault

vault.put

Store a content-addressed artifact.

Arguments:
  kind     (required)  "fact" | "event" | "decision" | "skill" | "profile" |
                       "tool_obs" | "summary" | "project"
  payload  (required)  JSON object — the artifact's content
  tags     (optional)  string array — structural labels (affect the hash)
  refs     (optional)  array of {kind, id} — references to other artifacts
  source   (optional)  {agent?, tool?, repo?, run_id?} — provenance (not hashed)

Returns: {id, created, created_at, last_seen_at}

vault.get

Retrieve an artifact by its SHA-256 ID.

Arguments:  id (required) — SHA-256 hex string
Returns:    Full artifact envelope, or {error: "NOT_FOUND"}

vault.search

Search artifacts with full-text, semantic, or hybrid scoring.

Arguments:
  query             (optional)  search text
  kind              (optional)  filter by artifact kind
  tags              (optional)  string array — AND logic
  exclude_personal  (optional)  filter out "personal:*" artifacts
  limit             (optional)  default 10
  offset            (optional)  default 0
  search_mode       (optional)  "hybrid" | "semantic" | "lexical" (default: "hybrid")

Returns: {total, offset, limit, search_mode, results: [{id, kind, score, tags, created_at, snippet}]}

vault.reindex_embeddings

Compute embeddings for all artifacts missing them.

Arguments:  batch_size (optional, default 32)
Returns:    {embedded, total, model, dim} or {error: "ENDPOINT_UNAVAILABLE"}

6. Search: Lexical, Semantic, and Hybrid

The vault supports three search modes via the search_mode parameter in vault.search.

Lexical (FTS5)

The default fallback. Uses SQLite’s FTS5 full-text search engine. Tokenizes your query, matches against payloads, tags, and kinds. Works without any external services.

Best for: exact keyword lookups, known terms.

Semantic (Embeddings)

Uses vector embeddings to find artifacts by meaning rather than exact words. Your query is embedded into a vector, and all candidates are compared by cosine similarity. Requires a running embedding endpoint.

Best for: fuzzy queries, concept-based search, questions.

Hybrid (FTS + Embeddings)

The default mode. Combines both approaches:

  1. A pool of FTS candidates is retrieved (5× the limit or at least 50).
  2. Each candidate is scored by cosine similarity against the query embedding.
  3. Final score = 60% FTS (normalized) + 40% cosine similarity.
  4. Results are ranked by the combined score.

Best for: general-purpose search balancing keyword precision with semantic understanding.

Fallback behavior: If the embedding endpoint is unavailable, hybrid and semantic modes automatically fall back to lexical search. You never get an error — you just get keyword-only results.

7. Embedding Setup

Embedding search requires a local server implementing the OpenAI /v1/embeddings API.

LM Studio

  1. Download LM Studio.
  2. Load an embedding model (e.g., nomic-embed-text-v1.5).
  3. Start the local server (default port: 1234).
  4. The system connects to http://localhost:1234/v1/embeddings by default.

Ollama

  1. Install Ollama and pull a model: ollama pull nomic-embed-text
  2. Run ollama serve.
  3. Set EMBEDDING_ENDPOINT=http://localhost:11434/v1/embeddings.

text-embeddings-inference (HuggingFace)

  1. Run the container with your model.
  2. Set EMBEDDING_ENDPOINT to the container’s URL.

Backfilling embeddings

After starting your embedding server, backfill existing artifacts:

vault.reindex_embeddings

New artifacts are embedded automatically on vault.put. Embedding is fire-and-forget — it never blocks or fails the put operation.

8. Progressive Memory

The memory system provides session-based conversation memory with progressive summarization. It keeps recent turns verbatim, compresses older turns into summaries, and distills the oldest summaries into extracted facts.

The three bands

BandAgeContentHow it is created
0Last 5 turnsVerbatimStored by memory.ingest_turn
1Turns 6–20SummariesCreated by memory.compact from Band 0
2Turns 20+FactsCreated by memory.compact from Band 1

Typical session workflow

  1. Start a session: Call memory.session_start.
  2. Ingest turns: After each conversation turn, call memory.ingest_turn.
  3. Compact periodically: Call memory.compact to age older turns into summaries and facts.
  4. Get context: Call memory.get_context with a token budget. Returns a context string with recent turns first, then summaries, then facts.
  5. End the session: Call memory.session_end. All artifacts remain in the vault.

Context format

[Turn 3] user: What is the weather today?
[Turn 4] assistant: The weather is sunny and 72F.

[Summary turns 0-2]: User discussed project setup and asked about dependencies...

[Extracted facts]:
- Project uses TypeScript with ESM modules
- Database is SQLite with FTS5

9. The Deterministic VM

The VM executes structured programs against a state machine with deterministic results. Same program + same state + same env = same output, always.

State structure

{
  bags:  { "energy": 100, "momentum": 50 }     // named numeric containers
  stack: []                                      // execution stack
  flags: { "threshold_met": true }               // boolean flags
  notes: ["Step completed"]                      // annotation strings
}

Environment

{
  run_seed:    42,       // seed for deterministic randomness
  world_seed:  7,        // world configuration seed
  max_steps:   10000,    // step limit before halt
  params:      {}        // additional parameters
}

Opcode families

FamilyOpcodesPurpose
Attract add, collect, select, increment Gather, collect, add values
Contain threshold, clamp, normalize, commit_to_stack, pop_from_stack Constrain, limit, validate
Release split, finalize, merge, conditional Split, output, finalize
Repel guard, negate, scatter, suppress Guard, reject, prevent
Transform map, smooth, exchange Modify, convert, exchange

Phase scanning

A phase scan sweeps over parameter ranges (knobs) and runs the program at every combination. It detects phase transitions:

  • Zero crossing — a metric crosses zero
  • Sign change — a metric changes sign
  • Threshold crossing — a metric crosses a threshold
  • Regime transition — halt behavior changes

Three scan modes:

  • grid — execute every point in the Cartesian product
  • adaptive — grid, then refine around transitions
  • hunt_boundaries — expand outward from boundaries, then bisect

Example program

{
  "program": {
    "program_id": "energy-distribution",
    "version": "program.v1",
    "opcodes": [
      {"opcode_id": "attract.add", "verb": "Attract",
       "args": {"bag": "pool", "amount": 100}},
      {"opcode_id": "release.split", "verb": "Release",
       "args": {"source": "pool", "targets": "a,b,c", "n": 3}},
      {"opcode_id": "contain.threshold", "verb": "Contain",
       "args": {"bag": "a", "threshold": 30, "flag": "a_ready"}}
    ]
  },
  "state": {"bags": {}, "stack": [], "flags": {}, "notes": []},
  "env": {"run_seed": 42, "world_seed": 1, "max_steps": 100}
}

11. Artifact Blessing and Promotion

What is blessing?

Blessing marks an artifact as verified and trustworthy with supporting evidence.

Blessing workflow

  1. Collect evidence: Use artifact.collect_evidence to gather refs from pipelines, executions, and artifacts.
  2. Bless the artifact: Use artifact.bless with the target hash, evidence refs, and a reason.
  3. Lifecycle transitions:
    • artifact.deprecate — mark as outdated
    • artifact.supersede — replace with a newer artifact

What is promotion?

Promotion extracts higher-order knowledge from raw imports:

  • promote_facts — extract factual claims from document snippets
  • promote_skills — extract procedural knowledge from execution evidence
  • promote_summary — create a summary spanning multiple sources
  • build_bundle — package all promoted artifacts into a single bundle

12. Bundles, Pinsets, and Behavior Packs

Bundles

A portable package of cards. Export to share knowledge, back up, or transfer between machines. Import verifies integrity hash and skips duplicates.

Pinsets

Named list of card IDs — like a bookmark folder. No policies, no search effect.

Behavior Packs

A pinset promoted with policies that actively change system behavior:

  • Search boost: Pinned cards get +500 points in search rankings.
  • Tag filtering: allowed_tags and blocked_tags control card visibility.
  • Export scope: pack_only exports just pinned cards; all exports everything.
  • Style: Visual style for rendered PNGs.

Only one pack active at a time.

16. Troubleshooting

“ENDPOINT_UNAVAILABLE” when using vault.search with semantic mode

The embedding endpoint is not running. Start your embedding server or use lexical mode. Hybrid mode falls back to lexical automatically.

“TierAccessError”

Your THREADFORGE_TIER does not have access to this tool. Set THREADFORGE_TIER=gold or use a tool available at your tier level.

“TierCapError: Artifact cap reached”

Your tier’s artifact limit is reached (Bronze: 1,000 / Silver: 10,000). Upgrade your tier. Re-putting existing artifacts (deduplication) does not count against the cap.

“Determinism violation” on vault.put

Your payload contains a prohibited key (timestamp, hostname, pid, cwd, __proto__, etc.). Remove these keys — they would make the hash non-deterministic.

Hash mismatch in Browse screen

The card’s content on disk does not match its hash. Possible causes: manual editing, different serialization, file corruption, or incompatible import.

TUI shows blank screen or crashes

Ensure Node.js 20+ and a standard terminal emulator. Try outside VS Code’s integrated terminal if it causes issues.

npm test fails with SyntaxError

Node.js 20+ required. Check with node --version.

Smoke test hash drift

Expected after intentional schema changes. Refresh the golden reference:

npm run smoke -- --refresh-golden
